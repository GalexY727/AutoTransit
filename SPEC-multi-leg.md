# Multi-Leg Transit Support — Design Spec
*Date: 2026-05-21 | Resolves: #2*

## Overview

Extend AutoTransit to handle itineraries with multiple transit legs (bus transfers). Single-leg trips are unchanged in output and code path. Multi-leg trips either produce one combined calendar event or two split events depending on transfer wait time.

---

## Data Extraction Layer

### `getTransitLegs_(itinerary)` *(new)*
Filters `itinerary.legs` to only `leg_mode === "transit"` entries. All other leg-extraction helpers delegate to this instead of duplicating the filter loop.

### `getRelevantBusTimes_(itinerary)` *(updated)*
- Was: `[depart, arrive]` (single pair)
- Now: `[[depart1, arrive1], [depart2, arrive2], …]` — one pair per transit leg, in order

### `getRelevantBusStops_(itinerary)` *(updated)*
- Was: `[onStop, offStop]` (single pair)
- Now: `[[on1, off1], [on2, off2], …]` — one pair per transit leg, same indexing as bus times

### `getBusNumber_(itinerary)` *(updated)*
- Was: first transit leg's `route_short_name`
- Now: all transit legs' route short names joined with `" → "` (e.g. `"19 → 16"`)
- Single-leg output is unchanged

### `getNextDeparture_(itinerary)` *(unchanged)*
Reads `departures[1].departure_time` from the first transit leg. Behavior is identical for multi-leg trips — the "next service" concept applies to the first boarding stop.

---

## Transfer Detection

Transfer time between leg N and leg N+1:
```
transferSeconds = leg[N+1].start_time − leg[N].end_time
```

**Split threshold:** 900 seconds (15 minutes). If *any* transfer in the itinerary meets or exceeds this threshold, use split mode. Otherwise use combined mode.

---

## Description Formats

### Single leg (1 transit leg)
Existing format, zero changes.

### Combined multi-leg (2+ legs, all transfers < 15 min)
One calendar event spanning the full itinerary (`itinerary.start_time` → `itinerary.end_time`).

```
Bus leaves in 5 minutes at 10:16 AM
 ➟ Last updated at 10:11 AM

Leg 1 — Route 19
Get on at:   Western Dr / Western Ct @ 10:16 AM
Get off at:  Downtown Transit Center @ 10:28 AM

Transfer: 8 min wait

Leg 2 — Route 16
Get on at:   Downtown Transit Center @ 10:36 AM
Get off at:  Bay / High (UCSC / Main Gate) @ 10:44 AM

Auto-generated by AutoTransit for:
ECE 10 @ UCSC Rachel Carson Academic 250

auto_commute_parent=<id>
```

- The `Bus leaves / Bus left` status line uses the **first** transit leg's departure time (same as current single-leg behavior).
- `getNextDeparture_` reads the first leg's `departures[1]` for the post-departure next-bus line.
- Transfer wait is rendered as `Transfer: N min wait` between leg blocks. The wait is computed as `leg[N+1].start_time − leg[N].end_time` in minutes (rounded), clamped to a minimum of 0 in case real-time data produces a negative gap.

### Split mode (any transfer ≥ 15 min)
Two independent calendar events. No combined event is created.

Each event uses the existing **single-leg description format** exactly, as if it were a standalone trip. Both footer lines reference the same parent class.

## Title Rules

| Mode | Format |
|---|---|
| Single leg | `🚍 19 to: ECE 10` (unchanged) |
| Combined multi-leg | `🚍 19 → 16 to: ECE 10` (all route nums joined with ` → `) |
| Split — non-last leg | `🚍 19 to: UCSC - Lower Campus` (alighting/transfer stop from `getRelevantBusStops_`) |
| Split — last leg | `🚍 16 to: ECE 10` (`parentEv.summary`, same as today) |

## Calendar Event Time Bounds (Split Mode)

- **Event 1:** `start = itinerary.start_time` (leave home, consistent with current), `end = leg1.end_time` (arrive at transfer stop)
- **Event 2:** `start = leg2.start_time` (board second bus), `end = itinerary.end_time` (arrive at class, consistent with current)

Event 1 starts when you walk out the door. Event 2 starts at bus 2 boarding time since there is no walk-to-stop context at a mid-trip transfer.

---

## Calendar Upsert Logic (Approach A — Sort-Match)

`upsertCommuteEvent_` gains an opening block that:
1. Calls `getTransitLegs_` to count legs and compute transfer times
2. Dispatches to one of three internal paths: **single**, **combined**, or **split**

### Single and Combined Paths
Use the existing upsert flow unchanged: find events by marker → patch `existing[0]` or create new. Combined just passes a richer description body.

### Split Path — Update Mechanism

1. Search for all existing events matching `auto_commute_parent=<id>` in a window spanning `[itinerary.start_time − 6h, itinerary.end_time + 6h]`
2. Sort hits by `start.dateTime` ascending
3. Zip sorted hits to legs: `hits[0]` → leg 1 event, `hits[1]` → leg 2 event
4. Patch matched events in place, create missing ones, delete any extras beyond 2

**Transition handling:**
| Before → After | Result |
|---|---|
| Single → Split | 1 existing hit: patch to leg 1, create leg 2 |
| Split → Single | 2 existing hits: patch `hits[0]` to single event, delete `hits[1]` |
| Split → Split | 2 existing hits: patch both in place |
| None → Any | 0 hits: create all needed events |

Both split events use the same `auto_commute_parent=<id>` marker (no leg suffix). The sort-match approach makes the marker unambiguous across transitions.

---

## What Is Not Changing

- `shouldProcess_`, `runPlanner` loop structure, `geocodeOrThrow_`, `pickBestItinerary_` — untouched
- Silent failure on API errors (try/catch in loop and in `transitPlanArriveBy_`)
- "Bus left at / Next departure" post-departure description logic
- Footer line format: `Auto-generated by AutoTransit for: <class> @ <location>`
- `dedent_`, `toRelativeTime_`, `formatLocation_`, `toQuery_`, `minsToMs_` — untouched
- The existing `COMMUTE_TAG_PREFIX` constant and marker format — unchanged

---

## Scope Boundary

This spec covers itineraries with exactly 2 transit legs. The logic generalises to N legs (the loop structure handles it naturally), but the split-event logic only applies when exactly 2 transit legs exist — if 3+ legs have a long transfer, the behaviour is combined-only. This edge case is rare given the UCSC transit context and can be revisited if needed.
